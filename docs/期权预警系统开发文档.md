# OptionSentry 开发文档

本文面向项目维护者与策略开发者。系统定位是期货期权行情分析与预警，不包含任何交易执行。

## 1. 架构目标

当前架构围绕四个边界设计：

1. 配置严格、版本化，不在生产代码中兼容旧字段。
2. 策略负责公式、条件编译和展示元数据，不负责数据源循环。
3. Runner 负责 live/backtest 调度、精确订阅和通知状态机。
4. GUI 从策略元数据生成表单，新增普通策略无需修改 GUI。

主链路如下：

~~~text
load_config()
  → build_strategy()
  → discover_universe()
  → apply_strategy_filter()
  → Strategy.compile()
  → StrategyCompilation.units
  → Runner execution planning
  → data_source.stream(exact_universe)
  → CompiledStrategy.evaluate()
  → AlertEngine.process()
  → Notifier
~~~

## 3. 配置系统

### 3.1 严格入口

根节点必须提供 schema_version = 1。允许的根表只有：

- runtime
- universe
- data_source
- strategies
- backtest
- notifications
- logging
- gui

每一层都会拒绝未知字段。旧的 datasource、notifier、price_basis、selected、threshold、扁平 min_value/max_value 等不会迁移或忽略。

最小合法配置：

~~~toml
schema_version = 1

[[strategies]]
id = "cp_default"
type = "cp_combo"
name = "CP组合预警"
enabled = true

[strategies.parameters]
min_value = 0.01
max_value = inf
~~~

### 3.2 策略身份

- type：策略实现类型，由注册表管理。
- id：配置实例的稳定身份，用于预警 key 和状态隔离，必须唯一。
- name：中文用户显示名。
- enabled：是否参与本次运行。
- parameters：由策略类声明并验证。
- filter：可选脚本和入口函数。

同一 type 可以出现多次。例如正向和负向 CP 组合使用两个 id、两组 parameters，但共享 cp_combo 实现。

### 3.3 凭据

TqSdk 和 SMTP 配置只保存 username_env/password_env，不接受明文凭据。GUI 登录输入只在当前进程中使用，并注入该进程环境；保存配置不会保存账号密码。

## 4. 策略契约

### 4.1 StrategyParameterSpec

Strategy.parameter_specs 是配置和 GUI 的共同来源，支持：

- float
- int
- bool
- string
- enum

字段 key、choices 使用英文稳定值；label、choice_labels 和 help_text 面向中文用户。

~~~python
parameter_specs = (
    StrategyParameterSpec(
        "direction",
        "方向",
        "enum",
        default="both",
        choices=("both", "call", "put"),
        choice_labels=("双向", "认购", "认沽"),
    ),
)
~~~

注册时检查 type/key 格式、重复参数、枚举选项和默认值。配置解析时检查缺失、未知、类型、范围和策略自定义跨字段规则。

### 4.2 Strategy

Strategy 子类定义：

- display_name、display_order；
- parameter_specs；
- 可选 metric_columns；
- validate_parameters()；
- compile(universe)；
- 可选 parse_key()、email_presentation()、row_background_color()。

from_config() 由基类统一把 StrategyConfig.parameters 注入具体 dataclass。

### 4.3 StrategyCompilation

compile() 返回 StrategyCompilation，其中包含多个 CompiledStrategy 单元。Compilation 汇总：

- strategy_id
- strategy_type
- name
- units
- condition_count
- required_symbols

一个常见单元边界是同一标的和到期月份。策略在启动期完成腿匹配与条件展开，避免每个行情周期重新组合。

### 4.4 CompiledStrategy

每个执行单元必须提供：

- condition_count
- required_symbols
- backtest_group
- data_requirements
- evaluate(snapshot, changed_symbols)

required_symbols 必须包含 evaluate() 读取的每个期权和标的。建议用 add_condition() 同时建立 condition_ids_by_symbol，再用 affected_condition_ids() 做增量评估。

预警 key 应使用策略实例 id，并包含分组、合约结构和影响状态的参数，不能使用可修改的中文 name 作为身份。

### 4.5 DataRequirements

DataRequirements 明确策略需要的行情能力：

- quote_fields
- kline_durations
- option_greeks

当前执行管线支持 last_price。声明未支持的字段、K 线或 Greeks 时，Runner 在订阅前给出明确 ConfigError，而不是静默运行出空结果。扩展行情能力时需要同时修改数据源、快照模型、Runner 和测试。

## 5. Runner 执行规划

AlertRunner.run() 的关键阶段：

1. 启动前验证所有筛选脚本。
2. 数据源发现完整候选 Universe。
3. 为每个策略应用独立筛选。
4. 调用 compile()，校验每个单元的 DataRequirements。
5. 合并所有 required_symbols，构造精确 stream_universe。
6. 根据数据源 mode 选择 live 或 backtest 规划。

### 5.1 Live

live 模式只建立一次精确订阅。每个 MarketSnapshot 到达后，所有执行单元根据 changed_symbols 只评估受影响条件。

### 5.2 Backtest

backtest 模式由 Runner 按 backtest_group 合并不同策略的单元。每组：

1. 合并组内 required_symbols；
2. 从 stream_universe 构造 group_universe；
3. 单独调用 data_source.stream(group_universe)；
4. 只评估该组的单元。

因此数据源不需要理解具体策略，也不会把无关到期月放入同一回测订阅。

## 6. TqSdk 数据源

discover_universe()：

1. 枚举活跃期权；
2. 应用 universe.mode：
   - all：按 exchanges 扫描；
   - include：先 include 后 exclude；
3. 批量查询合约元数据；
4. 解析标的期货；
5. 在 live 模式应用成交量和持仓量过滤；
6. 返回 Universe。

stream(universe) 只消费 Runner 提供的精确 Universe：

- live 使用 quote_subscription_batch_size 分批订阅最新价；
- backtest 使用 60 秒 K 线回放；
- 数据源暴露 mode，但不再按策略自行分组。

## 7. 内置策略

### 7.1 cp_combo

同一标的、到期、执行价的 call、put 与标的价格：

~~~text
deviation = call_price - put_price + strike - underlying_price
value = deviation / underlying_price
~~~

每个标的/到期生成一个执行单元。条件 required_symbols 包括 call、put、underlying。

### 7.2 abs_spread

同一标的、到期、期权方向的不同行权价两两组合：

~~~text
value = (first_price - second_price) / (first_strike - second_strike)
~~~

腿顺序按 CALL/PUT 规则规范化。required_symbols 包括两个期权和用于虚实度计算的 underlying。

两种策略都使用开区间判断：

~~~text
min_value < value < max_value
~~~

## 8. 策略筛选

配置结构：

~~~toml
[strategies.filter]
script = "user_filter_scripts/by_expire_rest_days.py"
entrypoint = "accept"
~~~

脚本入口：

~~~python
def accept(option, ctx) -> bool:
    return option.expire_rest_days is not None and option.expire_rest_days <= 60
~~~

筛选发生在编译前，只允许基于静态合约元数据。返回值必须是 bool。路径相对配置文件目录解析。

## 9. 预警与通知

ConditionEvaluation 进入 AlertEngine 后维护每个 key 的 active 状态：

- 默认只在 false → true 时产生 AlertEvent；
- alert_on_first_match = true 时首次 active 也告警；
- 离开区间会清除 active 状态。

Notifier 可组合文件与邮件。GUI 弹窗和声音由 MainWindow 消费同一 AlertEvent。通知失败会记录日志并继续行情循环。

## 10. GUI

GUI 可见标签、按钮、选项和提示优先中文；内部配置键与枚举值保持英文。

ConfigEditor 的策略表包含：启用、ID、类型、名称、参数、筛选脚本。StrategyEditDialog 根据 parameter_specs 动态创建控件：

- bool → 复选框；
- enum → 中文 choice_labels，itemData 保存英文 choice；
- int → 有界整数框；
- float/string → 文本输入。

新增普通策略不应在 GUI 增加 type 分支。策略类型显示名、参数表单、指标列和展示钩子均来自注册表。

## 11. 新增策略流程

Codex 或其他代码智能体必须使用 [.agents/skills/create-optionsentry-strategy/SKILL.md](../.agents/skills/create-optionsentry-strategy/SKILL.md)。核心检查：

1. 明确公式、分组、腿选择、数据需求和中文展示。
2. 在 strategy_types 下新增自注册模块。
3. 声明并验证 parameter_specs。
4. compile() 生成精确执行单元。
5. 覆盖公式、缺失行情、增量更新、required_symbols、backtest_group 和 GUI 元数据测试。
6. 不修改 config.py、GUI、Runner 或数据源来识别具体 type。

## 12. 测试、构建与发布

~~~powershell
uv run pytest -q
powershell -NoProfile -ExecutionPolicy Bypass -File .\package.ps1 -StopRunning
~~~

提交前执行 git diff --check，并检查旧配置字段是否重新进入生产代码或文档。发布时版本、标签和产物名称必须一致。

## 13. 已知边界

- 仅支持 TqSdk 数据源和 last_price 策略行情。
- 回测仅支持 60 秒 K 线。
- 账号权限决定可用交易所与历史行情。
- 不提供自动交易。
- 配置不兼容旧格式，升级应基于最新 config.example.toml 重建。
